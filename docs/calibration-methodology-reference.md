# 게이트 임계치 캘리브레이션 메서돌로지

이 문서는 `calibrate_gates.py`의 사용법과 결과 해석 가이드, 그리고 v1.1 잠정 기본값과 그 근거를 정리한다.

---

## 1. 캘리브레이션 범위와 한계

### 1.1 무엇을 캘리브레이션하는가

세 개의 게이트 임계치만 캘리브레이션 대상이다. 이들은 캐시된 시세 데이터만으로 즉시 평가 가능하므로 백테스트가 깔끔하다.

| 파라미터 | PRD 키 | v1.0 잠정값 |
|---|---|---|
| 가격 하락폭 | `gates.price.min_drawdown_pct` | 50 |
| MACD 윈도우 | `gates.macd.window_days` | 3 |
| 거래량 배수 | `gates.volume.multiplier` | 1.5 |

### 1.2 무엇을 캘리브레이션하지 않는가

**확장(enrichment) 단계의 점수 가중치는 이 단계에서 캘리브레이션하지 않는다.** 이유:

- 뉴스 분류 결과는 과거 시점에 존재하지 않는다 (LLM 분류를 소급 적용 가능하지만 비용이 비현실적).
- 펀더멘털 데이터는 발표 시점·정정 공시 등으로 look-ahead bias가 발생하기 쉽다.
- 주봉 MACD는 일봉 데이터에서 파생 가능하므로 이론상 가능하지만, 가중치 합리화는 운영 데이터가 더 풍부할 때 의미 있다.

확장 가중치는 **운영 6개월 이상의 알림 이력**을 바탕으로 v1.x에서 조정한다. 그 전까지는 PRD §5.5.1의 잠정값을 사용한다.

### 1.3 본질적 한계

- **생존자 편향(survivorship bias)**: 백필은 오늘(2026-05-05) 시점의 유니버스로 진행되므로, 이미 상장폐지된 종목들은 데이터에 없다. 실제 운영 시 매일 갱신되는 유니버스에서는 이 편향이 점차 줄어든다.
- **체제 변화(regime shift)**: 과거 5년 (2021~2026) 데이터로 캘리브레이션한 임계치가 미래 5년에도 최적이라는 보장은 없다. 분기 1회 재캘리브레이션을 권장 (R-07).
- **거래 비용 미반영**: 백테스트는 종가 기준의 이론 수익률이며 슬리피지·수수료는 미반영. 실제 매매 수익률은 그보다 낮다.

## 2. 워크플로우

```
1단계: 초기 백필 (PRD §11.5)
       ↓ (KR/US 5년치 일봉 SQLite로 적재)
2단계: 데이터 export
       ↓ (SQLite → parquet)
3단계: 캘리브레이션 백테스트
       ↓ (calibrate_gates.py 실행, OFAT)
4단계: 결과 해석
       ↓ (월평균 신호수, 90d 승률, Sharpe 비교)
5단계: v1.1 confirm
       ↓ (config.yaml 기본값 확정)
6단계: 분기 재캘리브레이션
       (체제 변화 추적)
```

### 2.1 데이터 export (2단계)

스크립트는 다음 컬럼의 parquet을 입력으로 받는다:

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `ticker` | str | 종목 코드 |
| `date` | datetime64 | 거래일 |
| `adj_close` | float | 수정종가 |
| `volume` | int64 | 거래량 |

SQLite에서 export하는 예:
```python
import sqlite3, pandas as pd

con = sqlite3.connect("screener.db")
df = pd.read_sql(
    "SELECT ticker, date, adj_close, volume FROM prices",
    con, parse_dates=["date"]
)
df.to_parquet("prices.parquet", index=False)
```

KR + US를 따로 캘리브레이션하려면 시장별로 export하여 각각 실행한다. 시장 특성이 다르므로 임계치도 다를 수 있다.

### 2.2 백테스트 실행 (3단계)

```bash
# 합성 데이터로 프레임워크 검증
python calibrate_gates.py --synthetic --output report_test.md

# 실제 KR 데이터
python calibrate_gates.py --data prices_kr.parquet --output report_kr.md

# 실제 US 데이터
python calibrate_gates.py --data prices_us.parquet --output report_us.md

# 그리드 서치 (OFAT 결과를 본 뒤 좁은 범위에서)
python calibrate_gates.py --data prices_kr.parquet --protocol grid --output report_kr_grid.md
```

OFAT(One-Factor-At-a-Time) 프로토콜은 한 파라미터씩 변경하면서 다른 파라미터는 기본값에 고정한다. 13회 백테스트 (base + 4×3 변동). 시장당 5~10분 소요 예상.

## 3. 결과 해석

### 3.1 핵심 지표

캘리브레이션 보고서의 비교표를 볼 때 다음 우선순위로 본다.

**1순위 — 월평균 신호수**

목표: **시장당 일 10~30 통과**, 월 환산 200~600건. 너무 많으면 LLM 비용과 알림 피로도 부담, 너무 적으면 시스템 가치가 떨어진다.

월평균 신호수는 게이트 통과 종목 수이지 알림 발송 수가 아니다. 점수 임계치(`scoring.alert_threshold`, 기본 60)에서 다시 5~10배 줄어든다고 가정하면, 게이트 통과 200~600건이 알림 30~120건/월 (일 1~5건)에 해당한다.

**2순위 — 90일 승률과 평균 수익률**

90일은 본 시스템의 주된 평가 horizon이다. 30일은 너무 짧아 노이즈가 크고, 180일은 의사결정과 거리가 멀다.

승률 50%는 동전 던지기 수준이다. 의미 있는 신호라면 **90일 승률 55~65%**, **평균 수익률 시장 대비 +3~+8%p** 정도는 나와야 한다.

승률이 50% 미만이면 해당 임계치 조합은 사용하지 않는다 (랜덤보다 못함).

**3순위 — Sharpe 유사 지표 (mean / std)**

같은 평균 수익률이면 변동성이 작은 쪽이 좋다. 본 시스템에서 의미 있는 값은 **0.3 이상**. 합성 데이터에서 0.1 부근이 나오는 건 예상된 결과 (랜덤 데이터엔 edge가 없음).

**최악 손실(min_180)도 같이 본다.** 평균이 좋아도 -50% 한 번 맞으면 운영상 부담이 크다.

### 3.2 trade-off 패턴

일반적으로 다음 trade-off가 관찰된다.

| 임계치 방향 | 신호수 | 승률 | Sharpe |
|---|---|---|---|
| 더 깊은 하락폭 (60, 70%) | 감소 | 증가 | 증가 |
| 더 짧은 MACD 윈도우 (1, 2일) | 감소 | 증가(보통) | 증가 |
| 더 높은 거래량 배수 (2x, 3x) | 감소 | 증가 | 증가 |

**예외**: 너무 극단적이면 (하락폭 80%+, MACD 윈도우 1일, 거래량 5x+) 신호수가 너무 적어 운영 의미 상실. 또한 -85% 이상 하락은 상폐 위험으로 평균 수익률이 오히려 낮아질 수 있음.

### 3.3 결정 기준

다음 두 단계로 결정한다.

**1. 후보 임계치 선정**: 월평균 신호수가 200~600 범위인 조합만 후보에 둔다.

**2. 후보 중 선택**: 후보들 사이에서 90d 승률 + Sharpe가 높은 쪽을 선택. 단, **신호수가 100 미만이면 통계적 신뢰도가 떨어지므로 제외**.

여러 조합이 비슷하면 보수적인 쪽 (= 임계치가 더 빡빡한 쪽) 을 선택. 운영 중 임계치를 풀기는 쉬워도 조이기는 어렵다 (이미 발송된 알림 회수 불가).

## 4. v1.1 잠정 기본값

실제 데이터 백테스트 전, 금융 이론과 실무 관행에 기반한 잠정값. 백필 후 캘리브레이션을 통해 v1.2로 확정한다.

### 4.1 `gates.price.min_drawdown_pct: 50`

**근거**: 일반 조정장에서의 하락폭은 −15~−25%, 약세장은 −30~−40% 수준. −50%는 단순 조정을 넘어 **개별 종목의 가치 재평가**가 진행 중인 영역으로, 평균회귀 전략의 고전적 진입 구간이다.

학술 연구에서 −50% drawdown은 mean-reversion이 통계적으로 유의미해지는 임계치로 자주 인용된다 (예: De Bondt & Thaler 1985 후속 연구들). 다만 −80%를 넘으면 가치 함정·상폐 위험이 급증하므로 PRD §5.3.1의 비선형 점수 곡선이 필요하다.

**대안**: 보수적이면 60, 공격적이면 40. 실제 캘리브레이션에서 KR과 US가 다르게 나올 수 있다 (US는 시장 효율이 높아 −50%가 더 드물 가능성).

### 4.2 `gates.macd.window_days: 3`

**근거**: MACD 0선 돌파나 시그널 라인 돌파는 일봉 종가 기준 1~2거래일에 확인 가능하지만, 1일은 일시적 노이즈를 포함할 위험이 있다 (다음 날 다시 음수로 회귀). 3일은 추세 확인과 신선도의 균형점.

7일 이상은 모멘텀 신호의 시의성이 떨어진다 — 진입 타이밍 측면에서 늦은 신호다.

**대안**: 빠른 진입 선호면 2, 신중한 진입 선호면 5.

### 4.3 `gates.volume.multiplier: 1.5`

**근거**: 거래량 평균의 1.5배는 의미 있는 관심 증가를 시사하면서도 너무 빡빡하지 않은 수준. 1.0은 사실상 필터가 아니고, 3.0 이상은 정상 추세 전환을 놓치고 작전·이벤트 종목만 잡힐 위험.

거래량 분포는 로그 정규에 가까우므로 1.5배는 분포의 상위 30% 정도에 해당한다.

**대안**: 노이즈 우려가 크면 2.0. 신호 빈도를 늘리고 싶으면 1.25.

### 4.4 종합 기대 효과

세 임계치를 위 잠정값으로 설정 시 예상 게이트 통과량 (KR 1500종목, 거래일 250일 기준):

| 게이트 단계 | 일평균 통과 | 비고 |
|---|---|---|
| Gate 1 (drawdown) | 100~200 | 시장 약세기엔 더 많음 |
| Gate 2 (MACD) | 15~30 | 약 15% 통과 |
| Gate 3 (volume) | 8~15 | 약 50% 통과 |

이 정도면 일 LLM 호출 50~100건 (게이트 통과 종목 × 종목당 5~10건 뉴스 / 2시장 합산), 월 LLM 비용 1~3만원 수준.

## 5. v1.1 확정 절차

1. 백필 워크플로우(`backfill.yml`)를 KR과 US에 대해 각각 실행 완료.
2. SQLite에서 `prices` 테이블을 시장별 parquet으로 export.
3. `python calibrate_gates.py --data prices_kr.parquet --output report_kr.md` 실행.
4. `python calibrate_gates.py --data prices_us.parquet --output report_us.md` 실행.
5. 보고서를 §3 기준으로 검토 → 시장별 권장 임계치 결정.
6. KR과 US가 의미 있게 다르면 `config.yaml`에 시장별 게이트 섹션 도입을 검토 (현재 PRD는 시장 공통 임계치).
7. v1.1로 확정 후 PRD 변경 이력에 기록 (실 데이터 백테스트 결과 첨부).

## 6. 후속 개선 (v2 백로그)

- **그리드 서치 자동 추천**: 보고서에 "권장 조합 Top 3" 자동 산출.
- **시장 지수 대비 알파**: 백테스트에 시장 인덱스(KOSPI, S&P500) 동일 기간 수익률 동봉.
- **섹터별 분포**: 신호 종목의 섹터 편중 여부 측정 (R-08 관련).
- **Walk-forward validation**: 단일 기간 백테스트가 아닌 롤링 윈도우로 임계치 안정성 검증.
- **트랜잭션 비용 모델**: 슬리피지·수수료 반영한 net 수익률 계산.
